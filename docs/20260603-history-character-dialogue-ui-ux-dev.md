# 历史人物对话界面与交互体验完善开发文档

## 1. 背景与目标

当前项目已经完成“虚拟历史人物对话”的核心链路：Next.js 前端提交历史人物和问题，FastAPI 后端调用 LangGraph Agent，经过 RAG 检索、LLM 生成、史实校验后返回回答、史料来源和校验状态。

本阶段目标不是重构底层 Agent，而是在现有能力基础上，把历史人物对话完善成一个更像真实教育产品的交互界面：学生能更自然地提问、看到清晰的生成过程、理解回答依据，并在失败时知道如何处理。

## 2. 当前项目现状

### 2.1 前端现状

位置：

- `frontend/app/page.tsx`
- `frontend/app/globals.css`

当前界面能力：

- 展示功能标题和说明。
- 提供历史人物快捷按钮。
- 支持输入历史人物和学生问题。
- 调用后端 `POST /api/history/character/chat`。
- 展示 Agent 返回的教学模拟回答。
- 展示史料依据卡片。
- 展示简单状态文案和错误提示。

当前交互不足：

- 页面仍是“表单 + 结果”的一次性问答形态，不像真实对话产品。
- 没有多轮消息区，学生难以连续追问。
- 加载过程只是一句状态文案，不能体现 Agent 检索、生成、校验流程。
- 人物选择只有按钮，没有人物卡片、推荐问题和教学标签。
- 史料依据只做平铺展示，缺少来源类型、命中主题、展开收起等阅读体验。
- 错误提示偏技术化，例如 `HTTP 500`，不适合学生和教师。

### 2.2 后端接口现状

位置：

- `backend/api/main.py`

当前接口：

```http
POST /api/history/character/chat
```

请求体：

```json
{
  "character": "商鞅",
  "message": "你为什么要变法？",
  "session_id": null,
  "grade": null,
  "stream": true
}
```

当前响应体：

```json
{
  "response": "同学你好，我将用“历史教学模拟”的方式，以商鞅的视角回答。...",
  "character": "商鞅",
  "sources": [
    {
      "topic": "春秋战国·商鞅变法",
      "source": "《战国策·秦策一》",
      "grade": "七年级上",
      "unit": "第三单元",
      "type": "primary",
      "content": "春秋战国·商鞅变法：商君治秦..."
    }
  ],
  "verified": true
}
```

当前接口可直接支撑本阶段 UI 优化，但后续如要实现真正多轮记忆和流式输出，需要继续扩展。

## 3. 本阶段开发范围

本阶段围绕之前建议的 5 个产品化方向进行界面和交互完善：

1. 统一前端展示结构，让页面从一次性表单升级为学习对话工作台。
2. 优化错误与状态反馈，让学生知道系统正在做什么、失败时如何处理。
3. 利用现有 `sources` 和 `verified` 字段强化史料依据展示。
4. 为多轮对话预留前端交互形态，但后端仍先按单轮请求工作。
5. 为后续流式展示保留 UI 结构，当前先使用 JSON 响应稳定实现。

不在本阶段实现：

- 真正的后端多轮会话存储。
- SSE/NDJSON 流式响应。
- 用户登录、班级、教师后台。
- 学生学习记录持久化。
- 大规模知识库管理后台。

## 5. 前端交互设计

### 5.1 人物选择体验

将当前简单 chips 升级为人物卡片。

推荐人物配置：

```ts
type CharacterPreset = {
  name: string;
  dynastyOrPeriod: string;
  tags: string[];
  defaultQuestion: string;
  suggestedQuestions: string[];
};
```

首批人物：

- 商鞅：变法、战国、制度改革。
- 秦始皇：统一六国、统一文字、中央集权。
- 唐太宗：贞观之治、治国策略。
- 林则徐：虎门销烟、鸦片战争。
- 孙中山：辛亥革命、民族民主革命。

点击人物卡片后：

- 自动填入 `character`。
- 自动填入默认问题。
- 展示该人物的推荐追问。
- 清空上一轮状态或提示“已切换人物”。

### 5.2 推荐问题

每个人物展示 3 个推荐问题，例如：

商鞅：

- 你为什么要变法？
- 变法对秦国有什么影响？
- 为什么变法会遇到阻力？

秦始皇：

- 统一文字为什么重要？
- 郡县制有什么作用？
- 统一六国后做了哪些措施？

林则徐：

- 虎门销烟有什么历史意义？
- 为什么清政府要禁烟？
- 鸦片输入带来了什么影响？

推荐问题点击后直接填入输入框，但不自动提交，避免学生误触。

### 5.3 对话主区域

把结果展示从“回答块”升级为对话流。

前端状态结构建议：

```ts
type ChatMessage = {
  id: string;
  role: "student" | "agent";
  character?: string;
  content: string;
  sources?: Source[];
  verified?: boolean;
  createdAt: number;
};
```

当前后端仍是单轮接口，所以前端可以先在本地维护消息列表：

1. 用户提交后，追加一条 `student` 消息。
2. 请求成功后，追加一条 `agent` 消息。
3. 切换人物时，可选择保留历史或提供“清空对话”按钮。

本阶段建议默认保留当前页面内的对话历史，不持久化。

### 5.4 Agent 过程反馈

当前后端没有流式事件，但前端可以基于请求状态展示固定步骤：

```text
1. 正在检索广东初中历史史料
2. 正在生成教学模拟回答
3. 正在进行史实校验
```

实现方式：

- 请求开始时显示步骤条。
- 在没有真实后端事件前，可使用前端本地状态按时间推进：
  - 0s：检索史料。
  - 0.8s：生成回答。
  - 2s：校验史实。
- 请求完成后全部标记完成。
- 请求失败后当前步骤标记失败。

注意：步骤文案只用于用户体验，不应声称真实后端进度已经完成某一步。

### 5.5 史料依据展示

基于当前 `sources` 字段，将史料卡片增强为可阅读组件。

每张卡片展示：

- 主题：`topic`
- 年级与单元：`grade` / `unit`
- 来源：`source`
- 类型：`type`
- 内容：`content`

类型展示建议：

```text
primary   -> 原始史料
secondary -> 教材/解释材料
unknown   -> 未标注
```

交互：

- 默认展示前 2 条史料。
- 超过 2 条时显示“展开更多史料”。
- 长史料内容默认截断，点击“展开全文”。
- 点击某条 Agent 回答时，右侧史料区显示该回答关联 sources。

### 5.6 校验状态展示

当前响应包含 `verified` 字段。

展示规则：

- `verified: true`：显示“已通过史实校验”。
- `verified: false`：显示“已生成，建议结合史料复核”。
- 请求失败：显示“本次未完成史实校验”。

文案要面向学生和教师，避免过度承诺“绝对正确”。

建议文案：

```text
已通过史实一致性检查：系统已根据检索到的史料对回答做过一次校验，但学习时仍建议结合教材理解。
```

## 6. 错误处理设计

当前前端错误是：

```text
请求失败：HTTP 500。请确认后端服务运行在 http://localhost:8000
```

建议改成分层提示。

### 6.1 前端错误映射

```ts
function getFriendlyErrorMessage(error: unknown) {
  if (error instanceof TypeError) {
    return "无法连接后端服务，请确认已在项目根目录运行 npm run dev。";
  }

  if (error instanceof Error && error.message.includes("HTTP 500")) {
    return "后端生成回答时出错，可能是模型 Key、模型服务或知识库加载异常。请查看后端终端日志。";
  }

  if (error instanceof Error && error.message.includes("HTTP 422")) {
    return "请求内容格式不正确，请检查历史人物和问题是否填写完整。";
  }

  return "请求失败，请稍后重试。";
}
```

### 6.2 页面提示方式

错误提示不要只放在状态行，建议使用明显的错误卡片：

- 错误标题：`暂时无法生成回答`
- 错误原因：友好错误文案
- 操作建议：
  - 检查后端是否运行。
  - 检查 `.env.local` 是否配置模型 Key。
  - 换一个人物或问题重试。

## 7. 输入体验优化

### 7.1 表单校验

前端提交前校验：

- 历史人物不能为空。
- 问题不能为空。
- 问题最少 2 个中文字符或 4 个英文字符。
- 问题最长建议 300 字。

提示文案：

```text
请先输入你想提问的历史人物。
请写下你的问题，例如：“你为什么要变法？”
问题有点长，建议拆成多个小问题继续追问。
```

### 7.2 快捷操作

建议增加：

- `清空对话`
- `复制回答`
- `换一个推荐问题`
- `只看史料依据`

复制回答只复制 Agent 回答文本，不复制完整 sources JSON。

## 8. API 配合建议

本阶段可以不改后端接口，但建议文档化前端依赖字段。

前端依赖字段：

```ts
type CharacterResponse = {
  response: string;
  character: string;
  sources: Source[];
  verified: boolean;
};
```

后续建议后端补充字段：

```json
{
  "response": "...",
  "character": "商鞅",
  "sources": [],
  "verified": true,
  "trace": {
    "retrieval_count": 5,
    "model_generate": "kimi-k2.5",
    "model_verify": "kimi-k2.6"
  },
  "suggested_followups": [
    "变法为什么会得罪旧贵族？",
    "商鞅变法和秦统一有什么关系？"
  ]
}
```

但 `trace` 和 `suggested_followups` 不作为本阶段必须项，避免扩大范围。

## 10. 验收标准

### 10.1 功能验收

- 可以选择商鞅、秦始皇、唐太宗、林则徐、孙中山等人物。
- 点击推荐问题能填入输入框。
- 提交后能看到学生问题气泡和 Agent 回答气泡。
- 回答完成后能看到史料依据。
- `verified: true` 时展示史实校验提示。
- 请求失败时展示学生可理解的错误提示。
- 可以清空当前页面对话。
- 可以复制 Agent 回答。

### 10.2 体验验收

- 页面不再像简单表单，而像一个历史学习对话工作台。
- 学生能理解系统不是“真实历史人物”，而是教学模拟助手。
- 学生能明确看到回答来自哪些史料。
- 教师演示时可以清楚讲解：检索史料、生成回答、史实校验三个 Agent 步骤。

### 10.3 技术验收

- `npm run build --prefix frontend` 通过。
- 根目录 `npm run dev` 可以同时启动前后端。
- 前端默认请求 `NEXT_PUBLIC_API_BASE_URL`，未配置时使用 `http://localhost:8000`。
- 不在前端暴露任何模型 Key。
- 不修改 `.env.local` 或提交敏感信息。

## 11. 后续迭代方向

### 11.1 真正多轮对话

后端使用 `session_id` 保存历史消息，让 Agent 能理解连续追问。

建议请求：

```json
{
  "session_id": "history-character-demo-001",
  "character": "商鞅",
  "message": "那为什么旧贵族会反对？"
}
```

### 11.2 流式输出

后端恢复流式响应，但建议使用更规范的 SSE：

```text
event: step
data: {"name":"retrieve","status":"done"}

event: delta
data: {"text":"同学你好..."}

event: sources
data: {"sources":[...]}

event: done
data: {"verified":true}
```

### 11.3 学习辅助能力

在回答后追加：

- 3 个追问建议。
- 1 道选择题。
- 1 个课堂讨论问题。
- 与教材单元对应的复习提示。

### 11.4 教师演示模式

提供“教师视图”：

- 展示 Agent 流程图。
- 展示检索命中的史料。
- 展示模型生成与校验结果。
- 适合课堂投屏讲解。
