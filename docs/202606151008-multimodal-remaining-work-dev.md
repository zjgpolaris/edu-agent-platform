# 多模态产品化剩余开发项文档

本文档承接 `docs/202606121040-multimodal-productization-roadmap-dev.md`，只记录当前多模态产品化路线中仍需继续开发、验证或补齐的工作。

当前判断：Level 1「资料上传学习 MVP」已完成并经过手动验证；后续开发重点应放在 Level 2 的资料 RAG 完整化、Level 3 的拍照批改产品化，以及 Level 4 的可追溯视觉引用。

## 1. 当前完成基线

### 1.1 已完成或基本完成

- 资料上传页面已存在：`frontend/app/material-upload/page.tsx`
- 资料解析接口已存在：`POST /api/materials/parse`
- 学习内容生成接口已存在：`POST /api/materials/analyze`
- 支持 PDF、PNG、JPG/JPEG 文件上传。
- 支持文本型 PDF 解析、图片 OCR、多模态转写回退。
- 支持 OCR 结果预览、编辑、校对确认。
- 支持生成知识点摘要、讲解和练习题。
- 支持将确认后的资料保存到个人资料库。
- 支持围绕已保存资料进行问答。
- 作业拍照批改已有基础页面和接口。

### 1.2 后续不再重复投入的范围

以下内容除非出现回归问题，否则不作为新开发重点：

- Level 1 上传解析基础链路。
- Level 1 摘要、讲解、练习题生成基础链路。
- OCR 低质量时要求用户校对确认的基础交互。
- 上传材料不直接写入全局历史知识库的隔离原则。

## 2. Level 2 剩余工作：资料 RAG 完整化

### 2.1 目标

把当前已经能保存和问答的资料库能力，补齐为更稳定、可管理、可验证的「个人/临时资料 RAG」功能。

### 2.2 待开发项

#### 2.2.1 保留真实页码与分页结构

当前保存资料时前端会把确认后的整段文本作为单页提交，容易丢失 PDF 原始页码信息。

待完成：

- 保存时优先使用解析接口返回的 `pages`。
- 用户编辑整篇文本后，仍尽量保留页码分隔标记。
- 后端保存 material 时按真实页码生成 chunk metadata。
- 问答来源展示真实页码，而不是统一显示第 1 页。

验收标准：

- 上传多页 PDF 后保存，问答来源能显示正确页码。
- 资料详情页能按页查看文本。
- chunk metadata 中包含 `material_id`、`owner_key`、`page_number`、`chunk_id`。

#### 2.2.2 资料列表与详情页完善

当前资料保存后主要停留在上传页内使用，缺少完整的资料管理体验。

待完成：

- 增加或完善「我的资料库」入口。
- 展示已保存资料列表：标题、学科、年级、来源类型、页数、chunk 数、创建时间。
- 支持进入资料详情页。
- 详情页展示：资料元信息、分页文本、处理提示、问答入口、删除入口。
- 删除前增加确认提示。

验收标准：

- 用户刷新页面后仍能找到已保存资料。
- 用户可以从资料列表进入某份资料并继续提问。
- 删除资料后，列表、数据库和向量库中均不可继续检索该资料。

#### 2.2.3 临时资料 TTL 与清理机制

当前 `materials` 表已经预留 `expires_at`，但缺少完整的过期策略和清理流程。

待完成：

- 区分「临时资料」和「长期保存资料」。
- 临时资料写入 `expires_at`。
- 增加清理函数，删除过期资料的 SQLite 记录和 Chroma 向量。
- 在保存接口或列表接口中触发轻量清理，或增加脚本/后台任务。

验收标准：

- 过期资料不会出现在资料列表中。
- 过期资料不能被问答接口访问。
- 过期资料对应向量被删除。

#### 2.2.4 资料 RAG 自动化验证

当前主要依赖手动验证，缺少稳定的 smoke/eval。

待完成：

- 增加一个资料 RAG smoke 脚本。
- 覆盖保存、检索、问答、删除链路。
- 使用小文本资料，避免依赖 OCR 环境。
- 验证 owner 隔离：用户 A 不能检索用户 B 的资料。

建议文件：

```text
eval/material_rag_smoke.py
```

验收标准：

- smoke 脚本能在本地一键运行。
- 删除资料后再次查询返回 404 或无结果。
- 不同 owner 的资料检索相互隔离。

## 3. Level 3 剩余工作：拍照批改产品化

### 3.1 目标

把当前已有的作业解析和批改基础能力，完善为可演示、可复用、能沉淀学生弱点的拍照批改闭环。

### 3.2 待开发项

#### 3.2.1 前端批改结果体验完善

当前已有 `frontend/app/homework-grading/page.tsx`，但仍需要强化结果呈现和可编辑体验。

待完成：

- 批改前支持逐题编辑题目、材料、学生答案、知识点标签。
- 批改结果按题展示分数、优点、问题、缺失要点、修改建议。
- 对 `needs_human_review` 做醒目提示。
- 展示追问练习 `follow_up_quiz`。
- 展示是否成功写入学生画像或弱点。

验收标准：

- 用户能完成「上传图片 → 校对题目/答案 → 批改 → 查看错因和复习题」完整流程。
- OCR 低置信度或学生答案为空时，必须要求人工确认。
- 批改结果中至少包含分数、错因、知识点、复习建议。

#### 3.2.2 增加选择题支持

当前后端仅允许 `history_short_answer` 和 `history_material_analysis`。

待完成：

- 扩展 `HomeworkTaskType`，增加 `history_single_choice` 或 `choice_question`。
- 作业抽取 prompt 支持选择题题干、选项、学生选择。
- 批改 prompt 支持判断选择是否正确并生成解析。
- 前端增加选择题任务类型。

验收标准：

- 上传一张选择题图片后，能抽取题干、选项和学生答案。
- 能判断正误并给出解析。
- 错题知识点能写入 weakpoints。

#### 3.2.3 与作文批改能力路由整合

路线图中 Level 3 包含作文批改，但当前作业拍照批改链路主要面向历史题。

待完成：

- 明确作文拍照批改是否进入 `homework-grading` 页面，还是跳转到现有作文批改页面。
- 如果进入统一页面，新增作文任务类型。
- OCR/转写后将作文正文路由到现有作文批改服务。
- 保留人工校对步骤，避免 OCR 错字直接影响评分。

验收标准：

- 用户能上传作文照片并校对文本。
- 校对后的作文能调用现有作文批改能力。
- 作文结果不与历史题批改结构混淆。

#### 3.2.4 学生画像写入结果可见化

后端已经在批改后尝试记录 learning event 和 weakpoint，但前端需要更明确地反馈。

待完成：

- 批改结果中展示 `event_id` 或「已写入学习记录」。
- 展示本次新增/命中的薄弱知识点。
- 从学生主页或弱点页面能看到拍照批改产生的弱点记录。

验收标准：

- 带 `student_id` 批改后，学生弱点页面能看到对应知识点。
- 不带 `student_id` 时，不显示已写入画像的误导提示。

#### 3.2.5 作业批改 smoke 验证

待完成：

- 增加作业批改 smoke 脚本或手动验证清单。
- 覆盖至少两类输入：正常题目、低置信/空答案。
- 验证 weakpoint 写入。

建议文件：

```text
eval/homework_grading_smoke.py
```

验收标准：

- 正常输入能得到结构化批改结果。
- 空答案或低置信输入会 `needs_human_review=true`。
- 有学生 ID 时能写入 learning event 和 weakpoint。

## 4. Level 4 剩余工作：可追溯视觉引用

### 4.1 目标

让资料问答不只展示文字来源，还能追溯到原始 PDF 页、图片区域或 OCR 文本块，提升可信度和展示效果。

### 4.2 待开发项

#### 4.2.1 保存原始文件或页图预览

当前资料解析后主要保存文本，没有完整保存原始 PDF/图片用于回看。

待完成：

- 设计上传文件存储策略：本地目录或对象存储。
- 保存 material 与原始文件的关联。
- 对 PDF 生成页图预览，或在前端支持 PDF 页查看。
- 控制访问权限，确保只能 owner 访问。

验收标准：

- 资料详情页可以打开原始图片或 PDF 页预览。
- 非 owner 无法访问文件。
- 删除 material 时同步删除原始文件或标记清理。

#### 4.2.2 OCR block 坐标结构

当前 OCR region 主要是文本分区，没有稳定的 bbox 坐标链路。

待完成：

- OCR 时保留 block 或 line 级 bbox。
- 数据结构增加 `block_id`、`page_number`、`text`、`bbox`。
- chunk metadata 记录关联的 `block_ids`。
- 多模态转写没有坐标时，需要明确 `bbox=null`，不能伪造位置。

验收标准：

- 图片 OCR 至少能返回文本块坐标。
- RAG source 能追溯到一个或多个 block。
- 坐标缺失时前端降级为文本片段引用。

#### 4.2.3 前端引用卡片与高亮

待完成：

- 问答结果中展示引用卡片。
- 引用卡片包含标题、页码、片段、相似度/检索方式。
- 点击引用后打开原图/PDF 页。
- 如果有 bbox，则在预览上绘制高亮框。
- 如果没有 bbox，则高亮文本片段或仅展示引用片段。

验收标准：

- 用户能从回答跳转到对应页。
- 至少支持图片资料的区域高亮。
- 页码或 bbox 不存在时，不显示虚假定位。

#### 4.2.4 引用真实性约束

待完成：

- 回答生成 prompt 明确禁止编造页码、片段和位置。
- 后端返回来源以检索结果为准，模型不得自行生成 source。
- 前端引用卡片只展示后端 `sources`，不解析模型回答中的伪引用。

验收标准：

- 当检索结果为空时，回答明确说明资料不足。
- 模型回答中出现未在 `sources` 中的页码时，前端不把它渲染为引用。

## 5. 推荐实施顺序

### 阶段 A：Level 2 收尾，先让资料库稳定可用

优先级最高：

1. 保存真实 `pages`，修复多页 PDF 页码丢失问题。
2. 增加资料列表/详情页。
3. 增加资料 RAG smoke。
4. 补齐删除与 owner 隔离验证。

完成后，资料上传能力会从「单页演示」升级为「可长期使用的个人资料库」。

### 阶段 B：Level 3 演示闭环

优先级中高：

1. 完善作业批改前端结果展示。
2. 明确并实现选择题支持。
3. 展示学生弱点写入结果。
4. 增加作业批改 smoke。

完成后，可以演示「上传作业 → 批改 → 弱点沉淀 → 后续复习」闭环。

### 阶段 C：Level 4 可视化可信来源

优先级中：

1. 设计原始文件存储。
2. OCR block bbox。
3. 引用卡片与图片/PDF 预览。
4. 高亮区域。

完成后，资料问答会具备更强的可信展示能力，适合作为作品集亮点。

## 7. 风险与边界

### 7.1 不要把 OCR 结果当成可靠事实

所有进入学习内容生成、资料入库和作业批改的 OCR/转写文本，都应保留人工校对步骤。低质量 OCR 必须提示用户确认。

### 7.2 不要污染全局知识库

上传资料继续进入独立 `materials` collection，不直接混入全局历史知识库。

### 7.3 不要伪造视觉引用

如果没有页码、原图或 bbox，就只能展示文本片段来源，不能假装支持区域高亮。

### 7.4 成本和性能限制

继续保留并完善以下限制：

- 单文件大小限制。
- PDF 页数限制。
- OCR/多模态转写超时处理。
- 单次生成文本长度限制。
- 向量入库 chunk 数限制。

## 8. 下一步建议

最建议先做 Milestone 1：资料库可用化。

原因：当前 Level 2 已经有后端基础，投入较小，但能明显提升产品完整度；同时它也是 Level 4 可追溯引用的前置基础。完成后再推进拍照批改和视觉引用，整体风险更低。
