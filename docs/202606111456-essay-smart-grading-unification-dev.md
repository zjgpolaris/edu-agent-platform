# 作文智能批改功能合并开发文档

## 1. 背景

当前首页同时展示两个与作文批改相关的功能入口：

- **作文批改仪表板**：教师端批量导入作文，集中查看批改结果、复核提醒与班级批改概览。
- **作文批改助手**：AI 辅助批改语文作文，提供多维度评分建议，支持史实一致性检查与教师复核提示。

两者都围绕“作文批改、评分建议、教师复核”展开，用户在首页会感知到较强重叠。为降低认知成本，建议将两者合并为一个一级功能：**作文智能批改**。

合并后，该功能内部通过模式或 Tab 区分：

- **单篇批改**：面向快速输入/上传一篇作文并获得 AI 评分与修改建议。
- **批量批改**：面向教师批量导入班级作文、查看批改进度、复核结果与班级统计。

## 2. 目标

### 2.1 产品目标

1. 首页只保留一个作文相关一级入口，减少功能重复感。
2. 用统一的“作文智能批改”承载单篇和批量两类场景。
3. 保留原有能力：AI 评分、修改建议、教师复核、CSV 导入、班级统计。
4. 强化教师工作流：导入作文 → AI 批改 → 复核 → 统计/导出。
5. 为后续扩展“学生自查作文”“历史论述题史实核查”“讲评课报告”预留结构。

### 2.2 技术目标

1. 前端合并两个功能卡片与页面入口。
2. 新增或调整统一页面路由，例如 `/chinese/essay-grading` 或 `/essay-grading`。
3. 后端复用现有 `/api/chinese/essay/grade` 能力，必要时扩展批量任务接口。
4. 批改结果统一数据结构，避免单篇与批量结果字段割裂。
5. 支持后续接入持久化数据库；MVP 可继续使用内存状态或本地临时任务记录。

## 5. 用户流程

### 5.1 单篇批改流程

1. 用户进入“作文智能批改”。
2. 默认展示“单篇批改”Tab。
3. 用户填写：
   - 作文标题
   - 年级/学段
   - 作文题目或写作要求
   - 作文正文
   - 可选：是否启用史实一致性检查
4. 点击“开始批改”。
5. 前端调用单篇批改接口。
6. 页面展示：
   - 总分与等级
   - 分项评分
   - 总评语
   - 优点
   - 修改建议
   - 风险/复核提示
7. 教师可点击“标记已复核”或“复制评语”。

### 5.2 批量批改流程

1. 用户切换到“批量批改”Tab。
2. 下载或查看 CSV 模板。
3. 上传 CSV 文件，字段建议包含：
   - 学生姓名
   - 学号/座号
   - 班级
   - 作文标题
   - 作文题目
   - 作文正文
4. 前端解析并预览导入数据。
5. 用户确认后点击“开始批量批改”。
6. 后端创建批量批改任务。
7. 页面展示任务进度：
   - 总数
   - 已完成
   - 待处理
   - 需复核
   - 失败数
8. 批改完成后展示结果列表。
9. 教师可筛选“需复核”作文并查看详情。
10. 支持导出结果 CSV。

## 7. 后端改造方案

### 7.1 现有能力复用

当前已有接口：

- `POST /api/chinese/essay/grade`

该接口可作为单篇批改基础能力。

### 7.2 推荐新增接口

#### 单篇批改

如果现有接口字段满足需求，可保持不变。若需要统一结构，建议调整为：

```http
POST /api/chinese/essay/grade
```

请求：

```json
{
  "title": "我的历史感悟",
  "grade_level": "七年级",
  "prompt": "请围绕秦统一的影响写一篇短文",
  "content": "作文正文...",
  "enable_fact_check": true
}
```

响应：

```json
{
  "id": "essay_xxx",
  "score": 86,
  "level": "良好",
  "summary": "文章结构完整，观点明确...",
  "strengths": ["观点清晰", "材料使用较准确"],
  "suggestions": ["可以补充更具体的史实", "结尾升华不足"],
  "rubric_scores": [
    {
      "name": "立意",
      "score": 18,
      "max_score": 20,
      "comment": "中心较明确"
    }
  ],
  "review_required": true,
  "review_reasons": ["涉及史实判断，建议教师复核"],
  "fact_check": {
    "enabled": true,
    "passed": false,
    "issues": ["文中关于秦朝制度的表述需要核实"]
  }
}
```

#### 批量任务创建

```http
POST /api/chinese/essay/batch-grade
```

请求：

```json
{
  "class_name": "七年级一班",
  "assignment_title": "历史小论文",
  "enable_fact_check": true,
  "essays": [
    {
      "student_name": "张三",
      "student_id": "01",
      "title": "秦统一的意义",
      "prompt": "请围绕秦统一写一篇短文",
      "content": "作文正文..."
    }
  ]
}
```

响应：

```json
{
  "task_id": "batch_xxx",
  "total": 30,
  "status": "queued"
}
```

#### 批量任务状态

```http
GET /api/chinese/essay/batch-grade/{task_id}
```

响应：

```json
{
  "task_id": "batch_xxx",
  "status": "running",
  "total": 30,
  "completed": 12,
  "failed": 0,
  "review_required": 3,
  "results": []
}
```

#### 批量结果导出

```http
GET /api/chinese/essay/batch-grade/{task_id}/export.csv
```

MVP 可先前端生成 CSV，后端导出接口可延后。

### 7.3 后端实现注意点

1. 批量批改应限制单次作文数量，避免长时间阻塞。
2. MVP 可串行调用单篇批改逻辑；后续再改成后台队列。
3. 批量任务状态如果先用内存保存，需要在页面提示刷新后可能丢失。
4. 批量任务应记录失败项，避免单篇失败导致整个任务失败。
5. 教师复核字段不应只依赖总分，应该结合：
   - 分数过低或过高
   - 模型置信不足
   - 史实一致性问题
   - 作文正文过短
   - 内容疑似跑题

## 8. 数据与评分结构

### 8.1 评分维度建议

语文作文默认维度：

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 立意与主题 | 20 | 中心是否明确，观点是否积极健康 |
| 内容与材料 | 25 | 内容是否充实，材料是否贴合作文要求 |
| 结构与逻辑 | 20 | 段落安排、起承转合、逻辑连贯性 |
| 语言表达 | 25 | 用词、句式、文采、表达准确性 |
| 书写/规范 | 10 | 标点、错别字、格式规范；纯文本场景可弱化 |

历史论述类作文可增加：

| 维度 | 分值 | 说明 |
| --- | --- | --- |
| 史实准确性 | 20 | 史实、时间、人物、制度表述是否准确 |
| 论证质量 | 20 | 是否能用史实支撑观点 |

### 8.2 复核规则建议

触发教师复核的情况：

1. `fact_check.passed = false`
2. 总分低于 60 或高于 95
3. 作文正文少于指定字数
4. 修改建议中包含“可能跑题”“史实不准确”“内容缺失”等风险标签
5. LLM 输出结构不完整，经过后端兜底修复

## 12. 风险与取舍

### 12.1 批量批改耗时较长

风险：一次批量导入几十篇作文，LLM 串行处理会等待较久。

MVP 处理：

- 限制一次最多导入 10～20 篇。
- 前端展示进度。
- 单篇失败继续处理下一篇。

后续优化：

- 后端任务队列。
- SSE 推送进度。
- 持久化批改任务。

### 12.2 AI 分数可信度问题

风险：教师可能直接采纳 AI 分数。

MVP 处理：

- 明确标注“AI 建议分”。
- 对高风险作文提示教师复核。
- 展示分项理由而不是只给总分。

### 12.3 单篇与批量 UI 复杂度

风险：一个页面承载太多能力，首屏变复杂。

MVP 处理：

- 默认只展示“单篇批改”。
- 批量能力放入第二个 Tab。
- 复核和统计先作为批量结果的一部分，不单独做复杂页面。

## 13. 推荐 MVP 范围

第一版建议只做：

1. 首页合并入口。
2. 新建统一页面。
3. 单篇批改接入现有接口。
4. 批量 CSV 导入预览。
5. 批量调用单篇接口并展示结果表格。
6. 基础统计：平均分、需复核数量、完成数量。

暂缓：

- 后端持久化批量任务。
- SSE 实时进度。
- 复杂班级维度统计。
- 多教师协作复核。
- 学生端提交与教师端作业流转。

## 14. 后续扩展

1. 学生端“作文自查”：学生提交后获得修改建议，但不展示正式分数。
2. 教师端“讲评报告”：根据全班结果生成课堂讲评提纲。
3. 历史论述题模式：强化史实一致性检查与史料引用建议。
4. 作文成长档案：按学生沉淀历次作文表现。
5. 批改标准配置：教师可自定义评分维度与分值。
