# 多模态产品化四层级规划

本文档基于当前 EduAgent 项目的已有能力，规划从“PDF/图片解析”到“可追溯多模态 RAG”的产品化路径。目标不是单独展示 OCR 技术，而是把多模态输入真正接入教育 Agent 的学习、练习、批改、检索和学情分析闭环。

## 1. 背景与目标

当前项目已经具备以下基础：

- 后端已有 FastAPI、SSE、认证、审计、限流和学习助手接口。
- 已有历史知识库 RAG、Chroma 向量库、Hybrid Search、Rerank 和来源展示能力。
- 已有教材结构化 YAML、教材问答、知识点总结、测验生成能力。
- 已有作文批改、历史人物对话、历史游戏和学生画像/弱点分析能力。
- 已有离线 PDF/OCR 相关脚本和依赖，如 `PyMuPDF`、`pytesseract`、`Pillow`。

多模态产品化的目标是：

1. 支持学生或教师上传教材 PDF、图片、截图、作业照片等材料。
2. 将上传内容解析为可确认、可编辑、可检索、可追溯的学习资料。
3. 复用现有 RAG、教材学习、测验生成、作文批改和学生画像能力。
4. 最终形成“上传材料 → 识别理解 → 学习交互 → 练习批改 → 弱点沉淀”的教育 Agent 闭环。

## 2. 产品化路线总览

多模态能力建议分四个层级推进：

| 层级 | 名称 | 目标 | 建议优先级 |
|---|---|---|---|
| Level 1 | 上传解析与学习材料生成 | 支持 PDF/图片上传，提取文本，生成总结和练习 | 最高 |
| Level 2 | OCR 内容入库与 RAG 问答 | 将上传材料写入知识库，支持基于资料问答 | 高 |
| Level 3 | 作业/试卷拍照批改 | 识别题目和答案，给出批改、错因和复习建议 | 中高 |
| Level 4 | 可追溯视觉引用 | 回答可定位到 PDF 页码、图片区域和 OCR 坐标 | 中 |

建议先完成 Level 1，形成可演示闭环，再逐步扩展到 Level 2 和 Level 3。Level 4 更适合作为高级展示能力，不建议一开始就作为 MVP 范围。

## 3. Level 1：上传解析与学习材料生成

### 3.1 用户场景

学生或教师上传一份教材 PDF、课堂讲义图片、历史材料截图，系统自动识别文本，并生成：

- 主要知识点。
- 简明讲解。
- 课堂提问。
- 3 到 5 道练习题。
- 可复制的学习笔记。

### 3.2 核心流程

1. 用户进入“资料上传学习”页面。
2. 上传 PDF、PNG、JPG 或截图。
3. 后端解析文件：
   - PDF：使用 `PyMuPDF` 提取文本。
   - 图片：使用 `pytesseract` OCR。
4. 前端展示识别结果。
5. 用户可手动修正 OCR 文本。
6. 用户点击：
   - “生成知识点总结”。
   - “生成练习题”。
   - “生成讲解”。
7. 后端调用现有 LLM/结构化输出能力生成结果。
8. 前端以卡片形式展示讲解、知识点、练习题和答案。

### 3.3 建议页面

新增页面：

```text
frontend/app/material-upload/page.tsx
```

页面模块：

- 文件上传区。
- 文件解析状态。
- OCR/解析文本预览区。
- 文本编辑确认区。
- 生成操作按钮。
- 学习结果展示区。
- 错误状态与重试入口。

### 3.4 建议后端接口

新增接口建议：

```text
POST /api/materials/parse
POST /api/materials/generate-summary
POST /api/materials/generate-quiz
POST /api/materials/generate-explanation
```

也可以先合并为一个 MVP 接口：

```text
POST /api/materials/analyze
```

请求包括：

- 文件。
- 学段/年级。
- 学科。
- 可选教材 ID。
- 任务类型：summary / quiz / explanation。

返回包括：

- 解析文本。
- 页码或图片序号。
- 识别置信信息（如果可用）。
- 生成的学习内容。
- 错误信息。

### 3.5 复用现有能力

可复用：

- `backend/llm_config.py` 的模型路由和流式输出。
- `backend/structured_output.py` 的结构化 JSON 输出。
- 教材学习中的总结、测验生成 prompt 思路。
- 学习助手页面中的 SSE 与状态展示模式。

### 3.6 MVP 验收标准

Level 1 完成时应满足：

- 能上传 PDF 和图片。
- 能提取可读文本。
- 用户能确认/编辑识别文本。
- 能生成知识点总结。
- 能生成不少于 3 道练习题。
- OCR/解析失败时有清晰错误提示。
- 不入库也能完成一次完整学习任务。

## 4. Level 2：OCR 内容入库与 RAG 问答

### 4.1 用户场景

教师上传一份课堂讲义或补充史料，系统将其加入临时或个人知识库。之后学生可以围绕这份材料提问，系统基于上传内容回答，并展示引用来源。

### 4.2 核心流程

1. 用户上传 PDF/图片。
2. 系统解析文本并按页/段落切分。
3. 用户确认资料标题、学科、年级、标签。
4. 后端将文本转为统一文档结构。
5. 文本被 chunk、embedding，并写入向量库。
6. 用户在资料详情页或学习助手中提问。
7. RAG 检索上传材料。
8. 模型基于检索片段回答，并展示来源。

### 4.3 知识库类型建议

建议先区分两类知识库：

1. **临时资料库**
   - 适合一次性上传和问答。
   - 可以设置 TTL。
   - 不影响全局历史知识库。

2. **教师资料库**
   - 适合教师长期维护。
   - 可关联班级、年级、教材章节。
   - 后续可加入权限控制。

### 4.4 数据结构建议

上传资料可抽象为：

```json
{
  "material_id": "string",
  "owner_id": "string",
  "title": "string",
  "subject": "history",
  "grade": "八年级上册",
  "source_type": "pdf|image|screenshot",
  "pages": [
    {
      "page_number": 1,
      "text": "...",
      "blocks": [
        {
          "block_id": "string",
          "text": "...",
          "bbox": null
        }
      ]
    }
  ],
  "created_at": "datetime"
}
```

RAG chunk metadata 建议保留：

- `material_id`
- `title`
- `owner_id`
- `grade`
- `subject`
- `page_number`
- `block_id`
- `source_type`

### 4.5 与现有 RAG 的关系

不建议一开始直接混入全局 `history` collection。更稳妥的做法是：

- 全局教材/史料继续使用现有 collection。
- 上传资料单独使用 `materials` collection。
- 检索时根据用户任务选择：
  - 只查上传资料。
  - 只查全局知识库。
  - 同时查两者并合并排序。

### 4.6 验收标准

Level 2 完成时应满足：

- 上传资料可以保存为 material。
- material 可以被 chunk 并写入向量库。
- 用户可以针对某份资料提问。
- 回答中展示来源页码或段落。
- 不同用户的资料不能互相检索。
- 可以删除或过期清理临时资料。

## 5. Level 3：作业/试卷拍照批改

### 5.1 用户场景

学生拍照上传历史问答题、选择题、材料分析题或作文，系统识别题目和学生答案，给出批改结果、错因分析、知识点关联和复习建议。

### 5.2 核心流程

1. 用户选择任务类型：
   - 历史问答题批改。
   - 材料分析题批改。
   - 选择题解析。
   - 作文批改。
2. 上传图片或 PDF。
3. OCR 提取题目和学生答案。
4. 前端让用户确认识别结果。
5. 后端根据类型路由：
   - 作文 → 现有作文批改能力。
   - 历史题 → 教材/RAG + LLM 批改。
   - 选择题 → 答案判断 + 解析生成。
6. 输出批改结果。
7. 将错题知识点写入学生画像和弱点分析。

### 5.3 题型识别建议

MVP 阶段不建议完全自动识别所有题型。可以让用户先选择：

- 作文。
- 历史简答题。
- 历史材料题。
- 选择题。

后续再加入自动分类。

### 5.4 输出结构建议

批改结果建议结构化：

```json
{
  "question": "题目文本",
  "student_answer": "学生答案",
  "score": 8,
  "max_score": 10,
  "grade_level": "良好",
  "strengths": ["..."],
  "issues": ["..."],
  "missing_points": ["..."],
  "related_topics": ["鸦片战争", "近代中国"],
  "revision_suggestion": "...",
  "follow_up_quiz": [
    {
      "question": "...",
      "answer": "..."
    }
  ]
}
```

### 5.5 与学生画像联动

批改后建议写入：

- 学生最近学习事件。
- 错误知识点。
- 薄弱主题。
- 推荐复习计划。

这样可以和现有：

- 学生 profile。
- weakpoints。
- review plan。
- learning assistant。

形成闭环。

### 5.6 验收标准

Level 3 完成时应满足：

- 能上传一张作业或试卷图片。
- 能识别题目和答案。
- 用户能修正 OCR 结果。
- 至少支持一种题型的批改闭环。
- 批改结果包含分数、错因、知识点和复习建议。
- 学生弱点数据能被更新。

## 6. Level 4：可追溯视觉引用

### 6.1 用户场景

学生问：“这段材料为什么说明洋务运动没有改变中国半殖民地化趋势？”系统回答时不仅给出文字解释，还能标注答案依据来自上传 PDF 的第几页、哪一段，甚至高亮原图区域。

### 6.2 核心能力

Level 4 需要在 Level 1 和 Level 2 的基础上保留更细粒度来源信息：

- PDF 页码。
- 图片文件 ID。
- OCR 文本块。
- 文本块坐标 bbox。
- 原图预览。
- 引用片段和答案句子的对应关系。

### 6.3 前端体验建议

回答结果中展示：

- 引用卡片。
- 页码信息。
- 原文片段。
- “查看原图”按钮。
- 图片/PDF 预览弹窗。
- 高亮引用区域。

### 6.4 技术实现建议

OCR 时保留 block 级信息：

```json
{
  "block_id": "page-1-block-3",
  "text": "洋务运动主张学习西方先进技术...",
  "bbox": {
    "x": 120,
    "y": 340,
    "width": 800,
    "height": 96
  },
  "page_number": 1
}
```

RAG chunk metadata 记录 block 关联：

```json
{
  "material_id": "...",
  "page_number": 1,
  "block_ids": ["page-1-block-3", "page-1-block-4"]
}
```

前端根据 metadata 回到原始图片或 PDF 页，并绘制高亮框。

### 6.5 验收标准

Level 4 完成时应满足：

- 回答来源可定位到页码。
- 用户可查看原图/PDF 页。
- 至少支持文本片段级高亮。
- 引用内容和回答结论能对应。
- 来源缺失时模型不能伪造引用。

## 7. 推荐实施顺序

### 阶段一：资料上传学习 MVP

优先完成：

- `POST /api/materials/parse`
- PDF/图片解析。
- OCR 文本预览和编辑。
- 知识点总结。
- 练习题生成。

目标：快速形成可演示闭环。

### 阶段二：上传资料临时知识库

继续完成：

- material 数据模型。
- `materials` Chroma collection。
- 基于 material 的问答。
- 来源页码展示。
- 用户隔离。

目标：把多模态内容接入 RAG。

### 阶段三：拍照批改

继续完成：

- 作业图片上传。
- 题目/答案确认界面。
- 单题型批改。
- 弱点写入。
- 复习建议生成。

目标：打通学习结果到学生画像。

### 阶段四：视觉可追溯

最后完成：

- OCR block 坐标。
- 原图/PDF 保存。
- 引用卡片。
- 高亮区域。

目标：提升可信度和展示效果。

## 8. 风险与注意事项

### 8.1 OCR 质量风险

OCR 对中文教材、拍照倾斜、手写字、低清图片可能不稳定。产品上必须允许用户确认和修正识别文本，不应直接把 OCR 结果当作可靠事实。

### 8.2 知识库污染风险

上传资料不应直接写入全局历史知识库。建议先使用临时 collection 或用户级 collection，避免错误 OCR 内容污染正式 RAG。

### 8.3 权限风险

上传材料可能包含学生姓名、班级、学校、成绩等敏感信息。需要考虑：

- 文件访问权限。
- 用户隔离。
- 删除机制。
- 审计日志。
- 敏感信息脱敏。

### 8.4 成本风险

PDF 页数过多时，OCR、chunk、embedding、总结都会带来成本和延迟。需要限制：

- 单文件大小。
- 页数。
- 图片数量。
- 单次生成 token。
- 后台任务超时。

### 8.5 产品复杂度风险

不要一开始就做完整多模态平台。建议先聚焦一个高频教育场景：

> 上传教材/讲义 → 生成知识点总结和练习题。

在这个场景稳定后，再扩展到 RAG 问答和拍照批改。

## 9. 面试与作品展示价值

完成四层级后，项目可以展示以下 AI Agent 工程能力：

- PDF/图片多模态输入。
- OCR 与文档解析。
- 文档结构化和 chunking。
- 向量检索与 RAG。
- 来源可追溯回答。
- 结构化输出。
- 多工具 Agent 工作流。
- 教育场景产品化交互。
- 学生画像和个性化复习。
- 评测、权限、安全和审计意识。

这比单纯做“上传图片让模型回答”更有工程含量，因为它把多模态能力接入了真实教育业务闭环。

## 10. 建议结论

当前最推荐的落地方案是：

> 先做“资料上传学习”MVP：支持 PDF/图片上传，提取文本，用户确认后生成知识点总结和练习题；随后再把确认后的材料写入临时 RAG 知识库，支持基于上传资料问答。

这个路径成本可控、复用现有项目能力最多、展示价值高，也最符合 AI Agent 工程师能力补齐目标。
